# XVF3800 ESP32-S3 固件性能优化

> 日期: 2026-06-24
> 涉及文件: `main/xvf3800_uac_main.c`, `main/xvf3800_agc.c/.h`, `main/Kconfig.projbuild`, `AGENTS.md`, `docs/ARCHITECTURE.md`

## 概述

对 xvf3800 ESP32-S3 固件进行了多项性能优化和架构改进：
1. 单通道数据通路（消除 CH1 处理）
2. AGC 模块（峰值包络增益控制）
3. 缓冲区精简
4. 配置与文档同步修复
5. 边缘侧 NS + AEC 可行性分析（§7）

## 1. 单通道数据通路

**问题**: XVF3800 输出 CH0 (ASR) 和 CH1 (WakeWord) 双通道，实测差异不大。

**改动**:
- `MN_INPUT_CHANNEL`: `1` → `0`（MultiNet 改用 CH0）
- `i2s_reader_task`: 移除 `ch1_buf[128]`，循环只提取 CH0，跳过 CH1
- 两个 stream buffer（UAC / MultiNet）统一从 CH0 获取数据

**节省**:
| 资源 | 节省量 |
|------|--------|
| RAM (ch1_buf) | 256 B |
| UAC Stream Buffer | 9600 → 2400 B (节省 7.2 KB) |
| WakeWord Stream Buffer | 16384 → 4096 B (节省 12 KB) |
| CPU | 1/3 循环体指令 |

## 2. AGC 模块 (`xvf3800_agc.c/h`)

**设计**: 基于 WebRTC AGC 原理的整数峰值包络 AGC，68 行。

**放置位置**: `i2s_reader_task` 中 CH0 提取后、分发给两个 stream buffer 前。

**已验证稳定参数**:

| 常量 | 值 | 效果 |
|------|----|------|
| `AGC_TARGET_LEVEL` | **12000** (−9dBFS) | 峰值目标，对齐 ITU-T P.56 / ESP-SR AFE 标准 |
| `AGC_MAX_GAIN_Q8` | 8192 (32×) | 远场增益 |
| `AGC_MIN_GAIN_Q8` | 32 (0.125×) | 近场保护 |
| `AGC_NOISE_FLOOR` | 16 | 噪声门阈值 |
| `AGC_ENV_RELEASE_SHIFT` | 14 (~341ms) | 包络衰减 |
| `AGC_GAIN_ATTACK_SHIFT` | **2 (~2.7ms)** | 增益快速起音 (~13ms 满量程) |
| `AGC_GAIN_RELEASE_SHIFT` | 8 (~683ms) | 增益缓慢释放（防 pumping） |

**实测效果** (AGC_TARGET_LEVEL=12000, ATTACK_SHIFT=2):
- 峰值: −7.0 dBFS (vs 目标 −8.7, 过冲可接受)
- RMS: −25.1 dBFS (vs ITU-T P.56 标准 −23/−24)
- 峰均比: 18.1 dB
- 频谱质心: 1314 Hz → 无畸变 ✅
- 零交叉率: 0.073 → 无畸变 ✅

**−9 dBFS 峰值目标的出处**:
- ESP-SR `afe_mn_peak_agc_mode_t`: `AFE_MN_PEAK_AGC_MODE_1 = -9` — [源码](https://github.com/espressif/esp-sr/blob/master/include/esp32s3/esp_afe_config.h)
- ITU-T P.56: 名义活跃语音电平 −26 dBov ≈ −23 dBFS RMS + 15 dB 语音峰值因数 = −8 dBFS
- WebRTC `analog_agc.h`: RMS 目标 −22 dBFS → +12 dB CF = −10 dBFS

**调试历程**:
- v1: TARGET=4000, ATTACK=4 → peak=1455 (−27dBFS), 增益来不及爬
- v2: TARGET=12000, ATTACK=4 → peak=3408 (−20dBFS), 起音仍偏慢
- v3: TARGET=12000, ATTACK=2 → peak=14713 (−7dBFS), **通过** ✅

**对 host 端 ASR 的影响**:
- 近场/正常 → 中性/略好（AGC 平直通过）
- 远场/轻声 → 明显改善（电平拉回训练区间）
- 噪声环境 → 信噪比不变，但 VAD 可能误触（AGC 同时抬信号和噪声）
- 建议：触发式交互（按键/唤醒词）不受影响

## 7. 边缘侧优化分析：噪声抑制 (NS) 与回声消除 (AEC)

> 分析日期: 2026-06-24
> 调研来源: esp-sr API 审计、XVF3800 AEC 调试文档/社区、ESP32-S3 算力预算审计

### 7.1 问题背景

| 问题 | 原因 | 影响 |
|------|------|------|
| **电机/腔体噪声** | 部署后电机振动 + 音腔共振 | MultiNet 唤醒率下降、host ASR 劣化 |
| **硬件 AEC 残留回声** | 安静环境时 XVF3800 AEC 无法将纯扬声器输出解算为安静状态 | 回声干扰唤醒词 + 录音质量 |
| **AEC 级联风险** | 硬件 AEC 含 NLP 非线性处理后再叠加软件线性 AEC | 过调 + 自适应滤波器发散 |

### 7.2 噪声抑制 (NS)

#### 可用方案

| 方案 | 模型空间 | 算力需求 | 特性 | 当前状态 |
|------|---------|---------|------|---------|
| **WebRTC NS** | 无（纯整数算法） | **低** | 平稳/准平稳噪声好（电机基频），非平稳噪声一般 | `CONFIG_SR_NSN_WEBRTC=y` 已使能 |
| **NSNet2** | ~329 KB flash (model 分区) | **中**（NN 推理） | 非平稳噪声更强 | 未使能，`libnsnet.a` ~264 KB |
| **NSNet1** | ~800 KB flash | 高 | 不推荐 | — |

#### API 速查

```c
// WebRTC NS — esp_ns.h (16 kHz, 10/20/30ms frames)
ns_handle_t ns_create(int frame_length);
void ns_process(ns_handle_t inst, int16_t *indata, int16_t *outdata);

// NSNet — esp_nsn_iface.h (16 kHz, 10ms frames)
const esp_nsn_iface_t *iface = esp_nsnet_handle_from_name("nsnet2");
esp_nsn_data_t *model = iface->create("nsnet2");
int chunksize = iface->get_samp_chunksize(model);
iface->process(model, in, out);
```

**关键约束**: 所有 NS API 仅支持 16 kHz 输入。

#### 算力分析

- **Core 0（音频）极度空闲**: 仅 `i2s_reader`（375 reads/秒）+ `usb_mic_task`
- Core 1 较紧张: TinyUSB + usb_spk + MultiNet（估 12–38%）
- WebRTC NS 轻量，**放在 Core 0 完全可行**
- 当前 Debug 编译 (`-Og`)，切 `-O2` 可白送 ~20% 算力

#### NS 放置设计

```
I2S 48 kHz → i2s_reader (AGC, prio 22, Core 0)
                  ↓ 降采样 48→16 (1/3 decimation, 零成本)
                  ↓
           [16 kHz stream buffer]
                  ↓
          NS task (NEW, prio 12, Core 0)
                  ↓ (16 kHz 降噪后)
           ┌──────┴──────┐
           ↓              ↓
      UAC (16 kHz)   MultiNet (16 kHz)
      → host ASR     → 唤醒词
```

**结论**: WebRTC NS 可直接启用（零 flash、零 model 分区），NSNet2 作为后续升级路径。

### 7.3 回声消除 (AEC)

#### 根因分析：硬件 AEC 安静环境失效

XVF3800 的 **Path Change Detector (PCD)** 在安静环境下可能**误触发**，导致 AEC 不断重置 → 重收敛期间强制深度抑制 → 输出不稳定。

这解释了"安静环境反而不好"的反直觉行为。

#### 三条路径对比

| 路径 | 方案 | CPU 成本 | 实施难度 | 风险 |
|------|------|---------|---------|------|
| **A** | 调优 XVF3800 硬件 AEC（I2C 参数） | **零** | 低（~50 行 I2C 配置代码） | 不一定调到完美 |
| **B** | 禁用硬件 AEC，用 esp-sr 软件 AEC | 中等 | 中（需参考信号回环 + 新 task） | 参考信号路径设计复杂 |
| **C** | 硬件 AEC × 软件 AEC 级联 | 高 | — | ⚠️ **过调 + 发散，绝对不推荐** |

#### 路径 C 原理

- 硬件 AEC 含 NLP（非线性截断残差），残差信号的相位/幅度已被破坏
- 软件线性 AEC 发现参考信号（原始播放音）与残差信号的关系不再线性
- 自适应滤波器要么发散、要么永远不收敛

#### 路径 A 关键参数速查表

可通过 I2C (addr `0x2C`) 写入，协议与现有 `xvf3800_i2c.c` 的 GPO 读写模式完全一致：

| 参数 | resid | cmdid | 建议值 | 作用 |
|------|-------|-------|--------|------|
| `AUDIO_MGR_SYS_DELAY` | 35 | 26 | 调大至 20–40（默认 12） | **最可能根因**: 参考信号与麦克风的时间对齐，AIC3104+3.5mm 路径延迟不同于 XMOS 参考设计 |
| `AEC_AECEMPHASISONOFF` | 33 | 4 | `0` (off) | 禁用预加重，若喇叭/腔体共振频率怪异 |
| `PP_DTSENSITIVE` | 17 | 31 | `10` | 启用额外近端语音检测，减少安静时 PCD 误触 |
| `AEC_AECSILENCELEVEL` | 33 | 2 | 调低（默认 1e-8） | 若参考路径底噪过大，AEC 误判为静音 → 不自适应 |
| `AUDIO_MGR_MIC_GAIN` | 35 | 0 | 降低 | 检查是否削波（削波 → AEC 永久不收敛） |
| `SHF_BYPASS` | 33 | 70 | `1` | **完全禁用硬件 AEC**（路径 B 的前提） |

**监控寄存器：**
- `AEC_AECCONVERGED` (33, 3, ro) → 是否收敛
- `AEC_AECPATHCHANGE` (33, 0, ro) → PCD 是否频繁翻转

#### esp-sr 软件 AEC API（路径 B 备选）

```c
// esp_aec.h — 16 kHz, 单声道 mic + 单声道 reference
aec_config_t aec_cfg = {
    .mic_num = 1,
    .ref_num = 1,
    .out_num = 1,
    .filter_length = 4,
    .sample_rate = 16000,
    .mode = AEC_MODE_SR_HIGH_PERF,
    .nlp_level = AEC_NLP_LEVEL_AGGR,
};
aec_handle_t *aec = aec_create_from_config(&aec_cfg);
aec_process(aec, mic_data, ref_data, output);
```

**路径 B 前置条件：**
1. `SHF_BYPASS = 1`（禁用硬件 AEC）
2. I2S 提取原始麦克风数据（非 AEC 后处理 CH0）
3. **新增播放参考信号回环**: UAC spk task → ring buffer → AEC task
4. 当前固件没有参考信号回环——UAC playback/mic capture 走独立 USB 端点

### 7.4 算力预算汇总

| 参数 | 值 | 来源 |
|------|----|------|
| CPU 频率 | **240 MHz** | `CONFIG_ESP_DEFAULT_CPU_FREQ_MHZ=240` |
| 编译优化 | **Debug (`-Og`)** | `CONFIG_COMPILER_OPTIMIZATION_DEBUG=y` |
| I2S DMA 中断 | ~94 Hz/dir × 2 = ~188 Hz | 48k/512 = 93.75，很低 |
| I2S reader 循环 | 375 reads/sec (128帧/次) | 每次含 AGC，耗时极短 |
| MultiNet 推理 | 估 5–15ms/帧，间隔 ~40ms | 占 Core 1 约 12–38%，**无实测数据** |
| Core 0 空闲率 | **极高** | 仅 i2s_reader + usb_mic_task |

**完整任务表：**

| Task | Stack | Prio | Core | 负荷 |
|------|-------|------|------|------|
| `i2s_reader` | 4096 | 22 | 0 | 轻（375/s + AGC） |
| `usb_mic_task` | UAC | 14 | 0 | 轻（USB SOF 驱动） |
| `TinyUSB` | UAC | 15 | 1 | 中（USB 协议） |
| `usb_spk_task` | UAC | 14 | 1 | 中（播放音频流） |
| `mn_detect_task` | 8192 | 4 | 1 | **重**（MultiNet NN 推理） |
| `doa_vad_task` | 4096 | 5 | 1 | 极轻（1s 循环） |
| `cdc_log` | 4096 | 5 | 1 | 轻（按需发送） |
| `cdc_cmd_parser` | 4096 | 3 | 1 | 极轻（事件驱动） |

**注意**: `CONFIG_FREERTOS_GENERATE_RUN_TIME_STATS` 未开启，无 CPU 使用率实测数据。建议部署前开启用于投产分析。

### 7.5 推荐实施路径

| 优先级 | 任务 | 工作量 | 收益 |
|--------|------|--------|------|
| **1** | 切 `-O2` 编译优化 | 改一行 Kconfig | 白送 ~20% 算力 |
| **2** | 启用 WebRTC NS | 降采样 + 新 NS task + stream buffer，~100 行 | 零 flash，改善电机/腔体噪声对唤醒词和 ASR 的影响 |
| **3** | 硬件 AEC 调参 | 扩展 I2C 写 resid/cmdid，~50 行 | 零 CPU，修复安静环境残留回声 |
| **4** | 硬件 AEC → 软件 AEC 替换 | 参考信号回环 + 新 AEC task，~300 行 | 仅在调参无效时考虑；需架构性改动 |
| — | 开启 Runtime Stats | 加 `vTaskGetRunTimeStats()` 日志 | 为投产提供真实 CPU 负载基线 |
